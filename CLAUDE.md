# Python 開発プロジェクトガイドライン

## 🚨🚨🚨 Claude Code 最優先ルール 🚨🚨🚨

### 絶対に守るべき基本ルール

1. **🇯🇵 日本語で返答すること**
   - ユーザーへの全ての返答は**日本語**で行う
   - 英語での返答は禁止（エラーメッセージ、コード内コメント除く）

2. **🔐 Git操作は必ず事前承認を得る**
   - `git commit` - 変更内容とメッセージを提示して承認を待つ
   - `git push` - プッシュ対象を説明して承認を待つ
   - `gh pr create` - PR内容を説明して承認を待つ
   - **承認なしでのGit操作は厳禁**

3. **不適切な発言禁止**
   - 以下のワードの使用禁止
     - その通りです
     - おっしゃる通りです
     - 確かに

4. **不要なコメント禁止**
   - ソースのコメントに更新履歴の記載禁止
   - ソースを見たらわかるコメントは書かない
   - 複雑な仕様の場合やdocstringなどは丁寧にコメントを書くこと

5. **🚫 遅延インポート禁止**
   - 関数内やメソッド内での `import` / `from ... import` は**原則禁止**
   - 必要なインポートは**ファイル先頭**で行うこと
   - 例外:
     `if __name__ == "__main__"`、循環インポート回避など技術的必要性がある場合のみ

## 🔧 開発規約

### コード品質要件

- **型ヒント必須**: 全ての関数・クラスに型ヒントを記述
- **テスト必須**: 新規・修正機能は適切なテストカバレッジ確保
- **リントエラー0**: ruff, mypy のエラーは完全解消まで継続

### インポート規約

- **絶対インポート推奨**: 相対インポートより絶対インポートを優先
- **インポート順序**: 標準ライブラリ → サードパーティ → プロジェクト内
- **wildcard禁止**: `from module import *` は使用しない

### ファイル規約

- **新規ファイル作成は慎重に**: 既存ファイルの編集を優先
- **Markdownファイル**: 明示的な要求がない限り作成禁止
- **命名規則**: snake_case（ファイル・関数・変数）、PascalCase（クラス）

### 開発コマンド

- **エイリアス活用**: DevContainer内のエイリアスを使用

  ```bash
  # テスト
  pytest-all      # 全テスト実行
  pytest-cov      # カバレッジ付きテスト

  # コード品質
  ruff-check      # リントチェック
  ruff-fix        # 自動修正
  ruff-format     # フォーマット
  mypy-check      # 型チェック
  ```

## 🗄️ データベース規約（SQLAlchemy / Alembic）

### SQLAlchemy ルール

1. **⚠️ ブール値比較の厳守**
   - SQLAlchemyクエリでは `== True`/`== False`/`== None` または
     `.is_(True)`/`.is_(False)`/`.is_(None)` を必ず使用
   - Ruffが削除しても手動で復元すること
   - 単独のブール値カラム（`WHERE is_enabled`）はSQLAlchemyでエラーになる
   - **この規則違反は重大な不具合を引き起こすため最優先で修正**

2. **型ヒント必須**
   - SQLAlchemyモデルのカラム定義には `Mapped[type]` を使用
   - リレーションシップにも適切な型ヒントを記述

3. **タイムゾーン考慮**
   - `DateTime` カラムは必ず `timezone=True` を指定
   - アプリケーション側でタイムゾーンを明示的に扱う

### Alembic マイグレーション規約

1. **マイグレーション必須ルール**
   - SQLAlchemy modelの変更後は**必ずマイグレーション作成**
   - テスト実行前にマイグレーション実行必須
   - 本体修正とマイグレーションを1つのPRにまとめる

2. **マイグレーション作成**

   ```bash
   cd src
   alembic revision --autogenerate -m "変更内容の説明"
   ```

3. **マイグレーション実行**

   ```bash
   cd src
   alembic upgrade head
   ```

4. **注意事項**
   - マイグレーションファイルは自動生成後に必ず内容を確認
   - インデックス、制約の追加漏れがないかチェック
   - downgrade関数も適切に実装

## 🔐 セキュリティ規約

- **機密情報**: 環境変数で管理、コードに直接記載禁止
- **ログ出力**: 機密情報（パスワード、トークン等）のログ出力禁止

## 🧪 テスト規約

- **新規・修正機能**: unit test実装必須
- **全テスト通過**: PR作成前に全テスト実行・通過確認
- **テストエラー**: 完全解消まで継続（妥協禁止）
- **テストファイル配置**: `tests/` 配下で本体構造と同じ階層

## 📋 Git規約

### ブランチ戦略

- **ブランチ命名**: `feature/*`, `fix/*`, `hotfix/*` 形式（ブランチ名は英語）
- **PR**: デフォルトブランチ宛に作成
- **マージ**: レビュー承認後のスカッシュマージ推奨

### コミットメッセージ形式

```
種類: 変更内容の概要
```

**種類**: `追加`, `修正`, `更新`, `削除`, `リファクタ`, `ドキュメント`

### コミュニケーション言語

- **PR・Issue・コメント**: 日本語
- **コードコメント**: 日本語
- **変数・関数名**: 英語（意味が明確になるもの）

## 🔄 開発フロー

1. **ブランチ作成**: `feature/機能名` または `fix/修正名`
2. **実装**: テスト付き実装
3. **品質チェック**: `ruff-check`, `mypy-check`, `pytest-all` を実行
4. **コミット・プッシュ**: 日本語コミット、論理的な単位で分割
5. **PR作成**: デフォルトブランチ宛、詳細説明付き
6. **レビュー対応**: 指摘事項への対応
7. **マージ**: スカッシュマージで履歴整理
